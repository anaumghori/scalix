## Ring Attention

Ring Attention is a distributed algorithm for computing transformer attention when the sequence length is too large to fit the full key and value tensors on a single GPU.  

Assume there are $P$ GPUs and the sequence is split evenly across them. Each GPU holds a shard of the queries, keys, and values corresponding to a subset of tokens. For example, if the sequence contains 16k tokens and there are four GPUs, each GPU may store 4k tokens. At the beginning of the computation each GPU can only compute attention between its local queries and its local keys and values. However, correct attention requires each query to interact with keys and values from all tokens in the sequence. To solve this, Ring Attention circulates the key and value shards between GPUs. The GPUs are logically arranged in a ring. During each step of the algorithm every GPU sends its current key and value shard to the next GPU in the ring and receives a shard from the previous GPU. After receiving the new shard, the GPU computes attention between its local queries and the received keys and values. The partial attention result is then merged with the previously accumulated results. This process repeats for $P$ steps so that every GPU eventually processes all key and value shards.  

The key idea is that queries remain stationary while keys and values move through the ring. Each GPU accumulates the attention contributions from every shard as the data circulates. At the end of the ring traversal each GPU has computed the complete attention output for the query tokens it owns, even though it never held the full sequence in memory at once.  

Because attention involves a softmax over the entire sequence, the algorithm must maintain numerically stable statistics while combining partial results. This is typically done using the `log-sum-exp` trick. Each step computes the local attention output and the `log-sum-exp normalization` for the current shard. These partial results are merged with previously accumulated values to produce the correct global softmax result.  

Ring Attention also allows communication to overlap with computation. While the GPU processes attention for the current shard, the next shard of keys and values can already be in transit across the network. When implemented carefully, this overlap hides much of the communication latency and allows the algorithm to scale efficiently across multiple GPUs.  


## Hierarchical collectives across nodes

A hierarchical collective organizes communication in stages that correspond to the hardware topology of the cluster. The system first performs communication inside each node and then performs communication between nodes. This structure reduces the amount of cross network traffic. Consider a cluster with eight nodes, where each node contains eight GPUs connected with NVLink. Suppose the training job runs on all sixty four GPUs using data parallelism and needs to perform an all reduce of gradients. Without hierarchy, the collective algorithm might treat all sixty four GPUs as a flat group. Communication messages would frequently cross machine boundaries, which means many messages would travel over the slower inter node network. 

**With hierarchical collectives, the operation proceeds in stages.**
1. First, an intra node reduction is performed. Each server reduces the gradients from its local GPUs. For example, the eight GPUs inside a node may perform an all reduce using NVLink. After this step each node has a single aggregated gradient representing the work of its local GPUs.
2. Second, an inter node reduction is performed. The aggregated gradients from the eight servers/nodes are reduced across the network using InfiniBand. Since there are now only eight participants instead of sixty four, the amount of network communication is much smaller.
3. Third, the result is distributed back inside each node so that all GPUs receive the final value.

This hierarchy matches the physical structure of the hardware. Fast GPU to GPU links handle most of the communication locally, while the slower network links carry fewer messages. The result is lower overall communication time.

## Gradient bucketing

Gradient bucketing is an optimization used in distributed training to improve communication efficiency and overlap communication with computation. During backpropagation, gradients are produced layer by layer. Launching a communication operation for every gradient tensor would create many small network messages, which is inefficient because each operation has overhead such as kernel launches, synchronization, and network latency. To reduce this overhead, gradients are grouped into larger buffers called buckets. As gradients are computed, they are placed into a bucket. When the bucket reaches a predefined size, a collective operation such as all reduce is launched on the entire bucket. This improves efficiency in two ways. Large messages utilize network bandwidth more effectively, allowing algorithms such as ring all reduce to achieve higher throughput. It also enables overlap between communication and computation. Since backpropagation computes gradients from the last layers toward earlier layers, communication for completed buckets can begin while gradients for earlier layers are still being computed. This hides communication latency and increases overall training throughput. 

Why this does not break the mathematics ?  Each parameter tensor has its own independent gradient reduction. The gradients of different parameters are not mathematically combined with each other. For example, if a model has parameters W1 and W2, the training system performs two independent reductions: Reduced gradient for W1 and then Reduced gradient for W2. Whether those gradients are transmitted separately or packed into the same bucket buffer does not affect the mathematical result. Buckets simply allow many independent gradient reductions to be transmitted together as one large communication operation.

## Designing a checkpoint system that scales to thousands of GPUs or recovers from partial failure.

Checkpointing is the process of periodically saving the training state so that training can resume after failures or interruptions. A checkpoint is essentially a snapshot of the full training state at a specific step. This state usually includes the model parameters, optimizer state, the current training step, random number generator states, and sometimes dataloader progress. Model parameters form the largest part of the checkpoint, and optimizer states can also be very large, especially for optimizers such as the Adam optimizer because they maintain additional tensors for each parameter. At the scale of thousands of GPUs, a checkpointing system must avoid communication bottlenecks, minimize pauses in training, and support recovery when failures affect only part of the cluster.  

A common design is distributed checkpointing. Instead of gathering all parameters onto a single machine, each GPU writes the portion of the model state that it owns. This matches how large models are typically partitioned in distributed training. In data parallel training, where each GPU holds a full copy of the model, systems often choose one GPU per node or group to write the checkpoint to avoid redundant storage. In sharded or model-parallel training methods such as ZeRO or Fully Sharded Data Parallel, parameters and optimizer states are already partitioned across GPUs, so each GPU writes its own shard directly to storage.  

To reduce training interruption, systems often use asynchronous checkpointing. Instead of blocking the training loop while data is written to disk, the training state is first copied into CPU memory or staging buffers. The checkpoint is then written to storage in the background while GPUs continue training. This approach reduces idle time and improves overall system utilization.  

Since a checkpoint may consist of thousands of files written by different processes, the system must maintain metadata that describes how the checkpoint is structured. The metadata records which shard belongs to which process, how parameters are partitioned, and the training step associated with the checkpoint. During recovery, this metadata allows each GPU to load the correct shard of the model and optimizer state.  

Failure handling is also important at large cluster scales. A common approach is coordinated restart, where the job manager restarts the training processes after a failure and reloads the most recent checkpoint so training can resume from the saved step. Some systems also support elastic or fault tolerant training, where the job can continue with a different number of workers if some nodes fail. In this case the checkpoint format must allow the model state to be redistributed across a different number of processes so that parameter shards can be reassigned correctly.  

Finally, the checkpoint system must guarantee consistency. If a crash occurs while a checkpoint is being written, the partially written checkpoint should not be considered valid. Many systems address this by writing checkpoint files to a temporary location and only marking the checkpoint as complete once all shards and metadata have been written successfully. This ensures that recovery always uses a consistent checkpoint.  

## Flex Attention

Flex Attention is best understood as a programmable attention framework at the kernel level. It exposes two primary control points through which the behavior of attention can be precisely customized.  

**1. Mask / Sparsity Pattern Function (block mask):** This defines which pairs (i, j) are allowed to interact, such as causal structure (attend only to past tokens), local windows (e.g., last 128 tokens), block-sparse layouts, or arbitrary structured connectivity. The core question it answers is: "Is interaction between i and j allowed at all?"  

**2. Score Modification Function (score_mod):** This defines how attention scores are computed or altered before the softmax. It operates on pairs of positions (i, j) and can depend on the query, key, or metadata, allowing users to add relative positional biases, incorporate learned or rule-based penalties, or encode structural preferences (e.g., favor nearby tokens). The core question it answers is: "Given that i can attend to j, how much should it prefer or penalize j?"  

The two roles are complementary. The block mask selects which tokens are even visible, and score_mod ranks among those visible tokens to determine which matter more. Consider a local window of size 128: the mask ensures tokens outside the window are impossible to attend to, but inside that window all 128 tokens are treated equally by default. Score modification lets you encode finer structure within that window.
